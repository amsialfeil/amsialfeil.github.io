# خطة التحسين الأمني — MagicToolsSDK v1.0.0

## مبنية على فحص فعلي للمشروع (تشغيل الكود، فحص C، اختراق عملي للتجربة)

## 🔴 المرحلة 1 — ثغرات حرجة (تُبطل نظام الترخيص بالكامل)

### 1.1 — تصفير التجربة بحذف ملف واحد
**المكان:** `protected_kernel.c:674-689`
**الوصف:** حذف `~/.d_law_trial` أو تعديل بايت واحد فيه يؤدي لإعادة إنشاء تجربة جديدة كاملة (30 يوم / 100 عملية) بدل رفض التشغيل. تم اختباره عملياً.
**الحل:** fail-closed — إذا الملف كان موجوداً سابقاً ثم اختفى أو تلف، النظام يقفل فوراً ولا يعيد الإنشاء. التمييز بين "أول تشغيل حقيقي" و"ملف محذوف بعد الإنشاء".

### 1.2 — بصمة الجهاز لا تعمل على Windows
**المكان:** `protected_kernel.c:371-427` — دالة `_get_device_fingerprint`
**الوصف:** تقرأ مسارات Linux فقط (`/sys/class/net/*/address`, `/proc/cpuinfo`, `/etc/machine-id`). على Windows تفشل كل محاولات القراءة ويعتمد فقط على `gethostname` — بصمة ضعيفة وسهلة التزوير.
**الحل:** إضافة مسارات Windows المكافئة (WMI، `GetAdaptersAddresses`، `GetComputerNameEx`، `HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Cryptography\MachineGuid`).

### 1.3 — الترخيص لا يستمر بعد إعادة التشغيل
**المكان:** `protected_kernel.c:636-637`
**الوصف:** `verify_license()` يفتح البوابة (`GATE_OPEN`) لكن لا يحفظ الحالة في ملف دائم. بعد إعادة تشغيل Python، `_gate_init()` يرجع لفحص التجربة فقط والرخصة المدفوعة تضيع — العميل يدفع ويخسر الترخيص بأول restart.
**الحل:** كتابة ملف ترخيص مشفر (AES-256-GCM) عند أول تفعيل ناجح، بحيث يستعيد `_gate_init()` حالة الترخيص عند كل تشغيل.

### 1.4 — باب خلفي hardcoded
**المكان:** `protected_kernel.c:796-804` — دالة `gate_force_open`
**الوصف:** الكود يحتوي على `DLAW_DEV_OVERRIDE=71A3F9C2` في النص الصريح. أي شخص يعرف هذا الرقم ويستطيع تشغيل `gate_force_open()` يتجاوز الترخيص بالكامل.
**الحل:** إزالة الباب الخلفي من كود الإنتاج. أو جعله يُقرأ من ملف خارجي غير موجود في الحزمة الموزعة.

## 🟠 المرحلة 2 — ثغرات عالية (تؤثر على موثوقية الحماية)

### 2.1 — فحص الترخيص ضعيف (HMAC فقط)
**المكان:** `protected_kernel.c:606-646`
**الوصف:** يعتمد على HMAC للبصمة فقط، لا يوجد توقيع رقمي حقيقي (RSA/ECDSA). مهاجم محترف باستخدام Ghidra/IDA يمكنه استخراج آلية التحقق وتوليد مفاتيح وهمية.
**الحل:** استخدام Ed25519 أو ECDSA للتوقيع الرقمي — المفتاح الخاص عندك فقط، والنواة تتحقق من التوقيع دون أن تمتلك القدرة على التوقيع.

### 2.2 — لا يوجد خادم ترخيص (offline بالكامل)
**الوصف:** لا آلية لإلغاء مفتاح مسرب أو تتبع عدد الأجهزة المفعّلة مركزيًا. أي عميل يمكنه نشر مفتاحه وآخرين يستخدمونه على أجهزتهم بدون علمك.
**الحل:** خادم تفعيل خفيف (serverless/API) يستقبل بصمة الجهاز + المفتاح ويعيد تأكيد الصلاحية. مع فترة سماح offline 7-14 يوم للعملاء الشرعيين.

### 2.3 — كشف التلاعب بالوقت متساهل
**المكان:** `protected_kernel.c:676`
**الوصف:** يسمح بانحراف 5 دقائق (`last_ts > now + 300`) قبل اكتشاف التلاعب. المستخدم يمكنه إرجاع الساعة قليلاً واستخدام النظام لأيام بدون اكتشاف.
**الحل:** تقليص السماح إلى 30 ثانية، مع تسجيل آخر طابع زمني موقّع داخلياً.

## 🟡 المرحلة 3 — مشاكل جودة واستقرار

### 3.1 — ملف fallback مفقود
**المكان:** `sdk/domain_template.py:29-31`
**الوصف:** يحاول قراءة `sdk/domain_templates.json` كحل احتياطي إذا فشلت النواة المحمية — لكن الملف غير موجود فعلياً في الحزمة. يؤدي لـ FileNotFoundError بدل رسالة خطأ واضحة.
**الحل:** تصدير المجالات إلى JSON ووضعه في الحزمة مع إضافته لـ MANIFEST.in.

### 3.2 — NaN/Infinity يمر بصمت
**المكان:** `protected_kernel.c:732-736`
**الوصف:** `divergence(nan, 12)` يرجع `nan` بصمت بدون خطأ. يتسرب NaN إلى نتائج التجميع دون تنبيه.
**الحل:** إضافة `isfinite()` في C وفي Law8Engine._validate_range في Python.

### 3.3 — لا توجد اختبارات آلية
**الوصف:** لا pytest، لا CI، لا تغطية. أي تعديل مستقبلي قد يكسر شيئاً دون اكتشافه.
**الحل:** إنشاء `tests/` مع تغطية لـ Law8Engine (aggregates)، DomainTemplate، وسلوك check_trial/verify_license كـ black-box.

### 3.4 — ضعف بصمة Windows (تكملة للـ 1.2)
**الوصف:** حتى بعد إصلاح قراءة العوامل، Windows لا يملك `machine-id` ثابت بنفس موثوقية Linux. الحلول البديلة تتطلب API أكثر عمقاً.
**الحل:** استخدام TPM (إن وُجد) أو ربط الترخيص بحساب Microsoft/Apple ID كطبقة إضافية.

## 🔵 المرحلة 4 — تنظيف وتحسينات (غير حرجة)

### 4.1 — مجلد build/ مكرر في الحزمة
**الوصف:** `build/lib/` يحتوي نسخة كاملة مكررة من sdk/ و protected/، يضاعف حجم الـ zip بدون فائدة.
**الحل:** استبعاده من ملف التوزيع النهائي. إضافته لـ .gitignore.

### 4.2 — معلومات شخصية في LICENSE.txt
**الوصف:** رقم هاتف وبريد شخصي يُرسلان لكل عميل داخل الحزمة.
**الحل:** بريد مخصص للمشروع (support@...) مستقبلاً.

### 4.3 — واجهة gate_status/gate_force_open غير موثقة
**الوصف:** مصدرة من `protected/__init__.py` لكن غير معاد تصديرها من `sdk/__init__.py`.
**الحل:** قرار واعٍ (إخفاء مقصود أمنياً أو سهو). التوثيق الصريح.

### 4.4 — مرساة ثانية للتجربة
**الوصف:** الاعتماد على ملف واحد فقط يجعل "النسخ والاستعادة" هجوماً ممكناً.
**الحل:** موقع تخزين ثانٍ؛ المقارنة بين المرساتين عند كل تحقق.
